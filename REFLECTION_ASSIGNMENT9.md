# Reflection
## Assignment 9: Domain Modeling and Class Diagram

---

## Introduction
This assignment focused on developing a domain model and class diagram for the Student Early Warning System. It required identifying key entities, defining relationships, and aligning system structure with previous assignments.

---

## Challenges Faced

One of the main challenges was abstraction. It was difficult to decide which entities should be included and how detailed they should be. For example, deciding whether RiskStatus should be part of the Student class or a separate entity required careful consideration.

Another challenge was defining relationships correctly. Understanding multiplicity (e.g., 1..*, 0..1) required thinking about real-world interactions between entities.

---

## Alignment with Previous Assignments

The class diagram aligns with:
- **Assignment 4**: Functional requirements such as risk detection and alerts
- **Assignment 6**: User stories such as monitoring students and providing intervention
- **Assignment 8**: State and activity diagrams showing system behavior

This ensured consistency across all stages of system design.

---

## Trade-offs

A key trade-off was simplifying the design. While more complex relationships could be modeled, a simpler structure was chosen to improve readability.

For example:
- Using separate classes instead of inheritance
- Avoiding overly complex relationships

---

## Lessons Learned

This assignment improved my understanding of object-oriented design and how system components interact. It also highlighted the importance of clear structure and consistency across diagrams.

---

## Conclusion

Overall, the domain model and class diagram provide a strong foundation for system implementation. They ensure that system structure aligns with requirements, workflows, and user needs.
