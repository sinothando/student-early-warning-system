# Reflection
## Assignment 6: Agile User Stories, Backlog, and Sprint Planning

---

## Introduction
This assignment provided valuable insight into applying Agile methodology to software development, particularly in translating system requirements and use cases into actionable user stories, backlog items, and sprint plans.

---

## Challenges in Prioritization
One of the main challenges encountered was prioritizing user stories effectively. Many of the features identified during earlier assignments appeared equally important, making it difficult to categorize them using the MoSCoW prioritization technique.

Deciding which features were **“Must-have”** versus **“Should-have”** required a deeper understanding of what constitutes a Minimum Viable Product (MVP). This process involved focusing on core system functionality, such as attendance tracking, grade management, and risk detection, while postponing less critical features like reporting.

---

## Challenges in Effort Estimation
Another significant challenge was estimating effort using story points. Since the system has not yet been fully implemented, assigning story points required assumptions about the complexity of each feature.

Without real development experience for this system, it was difficult to accurately estimate how long tasks would take. This highlighted the importance of experience and team collaboration in Agile environments, where estimates are usually refined through team discussion and iteration.

---

## Challenges of Working Individually
Working individually introduced additional challenges when applying Agile practices. Agile methodologies, such as Scrum, are designed for collaborative teams with distinct roles like Product Owner, Scrum Master, and Development Team.

In this assignment, these roles had to be performed by a single individual. This made decision-making more complex, as it required balancing multiple perspectives simultaneously. For example, prioritizing user-focused features while also considering technical constraints created internal conflict during planning.

---

## Traceability and Consistency
Another challenge was ensuring traceability between assignments. It was necessary to maintain consistency between stakeholder requirements (Assignment 4), use cases (Assignment 5), and user stories (Assignment 6).

Mapping these elements required careful attention to ensure that each user story could be traced back to a specific requirement and use case. This process emphasized the importance of structured documentation in software engineering.

---

## Lessons Learned
Despite these challenges, the assignment significantly improved my understanding of Agile principles and practices. It demonstrated how large system requirements can be broken down into smaller, manageable tasks and how iterative development supports continuous improvement.

---

## Conclusion
Overall, this assignment highlighted the importance of prioritization, estimation, and adaptability in Agile software development. It also reinforced the value of traceability and structured planning, even when working independently.
